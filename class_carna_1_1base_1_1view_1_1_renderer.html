Detailed Description

Represents the view component of a visualization and implements its rendering logic.

Attention:
You must not execute one and the same renderer within different GL contexts. You must assure that the contexts of all renderers do share their GL objects. The most simple way of accomplishing both requirements is using the qt::Display class.

This is an abstract base class. Implementations must provide definitions for renderColor and renderDepth. Providing a null-implementation for renderDepth might cause certain render modes, such as the Philips stereoscopic one, not to work properly.

Components

Each renderer instance is associated with a set of objects that determine the renderer's behavior. These objects and the supported types of their association are listed by the table below.

Object classAggregation supportedComposition supported
CameraXX
RenderModeX
Object3DPainterXX
SceneProviderX

Camera Management

The purpose of a camera is to define the coordinate mapping from scene into screen space. Refer to the illustration below for a better understanding. The camera may be aggregated (using Aggregation) as well as composed (using Composition) and is supplied via setCamera. Each renderer is instantiated with a DefaultCamera instance set initially.

See also:
Refer to the Integrated Logics section for details on the coordinate transformation pipeline.

Render Modes

The render modes concept enables 3D displays support. With the simple Monoscopic mode set by default, the 3D display support is disabled. It may be enabled via setRenderMode. Carna comes with two built-in stereoscopic render modes, which do enable support of certain 3D displays from Philips and Zalman.

3D Objects Management

The method paintObjects3D is responsible of rendering 3D objects. All it basically does is invoking the Object3DPainter, that can be supplied via setObject3DPainter. By default an instance of DefaultObject3DPainter is being used, that simply invokes the painting routine, that has been supplied to paintObjects3D, for each 3D object of the data model. If no painting routine is supplied explicitly, an instance of DefaultObject3DPaintingRoutine is used, that simply invokes the paint method of each object.

Data Model Interface

Finally, a reference to a SceneProvider object must be provided to the renderer's constructor, in order to specify the data model interface to use. The renderer registers itself as a client of the scene provider.

Integrated Logics

This class provides a simple shader manager. Refer to it's shader methods documentation for details.

Renderer implementations are allowed to provide any custom scene matrix, by overriding their fetchSceneMatrix method. The coordinate transformation pipeline is illustrated by the above figure. Refer to the Coordinate Systems page for a more detailed description.

virtual void Carna::base::view::Renderer::paint ( ) [virtual]

Runs all drawing operations.

See also:
The invalidated signal is emitted whenever the result, produced by this method, may have changed in relation to the result produced the last time the method was invoked.

The basic program flow is shown below:

 camera().openShutter( *this );

 renderMode().paint();

 camera().closeShutter( *this );

Nothing is drawn when the rendering lock is non zero. The rendering lock can be manually controlled via addRenderingLock and removeRenderingLock, which basically do increment and decrement a counter. The rendering lock is incremented implicitly when the paint routine catches an exception. So, if you want to proceed rendering, even if an exceptional state was raised, you would have to do something like this:

 try
 {
     renderer.paint();
 }
 catch( ... )
 {
     renderer.removeRenderingLock();
     throw;
 }

ShaderProgram& Carna::base::view::Renderer::shader ( const std::string &  vertex_shader_file,
const std::string &  fragment_shader_file 
) const

References the shader program, composed by specified vertex and fragment shaders.

Acquiring each shader on a per-renderer level would be a waste of resources. Since all renderers are assumed to be sharing their GL objects, the underlying implementation only acquires the shaders if they yet haven't been acquired by any other renderer. The shaders do stay acquired until the last renderer is destroyed - the size of shader objects is magnitudes smaller than the size of the volume data's 3D texture, therefore this weakness in memory management is negligible.
